# 企业知识管理系统（KMS）任务说明

最后更新：2026-04-19

## 1. 这份文档的用途

这份文档只承担两件事：

- 说明项目的稳定规则、当前真实边界和 AI 执行约束
- 维护当前待办事项的优先级与验收标准

不再在这里堆叠长历史日志。历史变化以 git 和 `进度.md` 为准。

## 2. 文档优先级

发生冲突时，按下面顺序理解项目：

1. 最新用户明确指令
2. `docs/安全基线.md`
3. 本文件
4. `进度.md`
5. 代码中的实际实现

## 3. 项目目标

本项目是企业知识管理系统（KMS）MVP，目标是交付以下核心能力：

- 用户登录与权限控制
- 知识仓库、目录、笔记、附件管理
- PDF / DOCX 入库与受控访问
- 全文检索与筛选
- 基于权限过滤的问答
- 管理后台
- Docker 化本地联调

## 4. 当前真实技术栈

- 前端：Next.js App Router
- 后端：FastAPI
- 数据库：MySQL 8.4
- 对象存储：MinIO
- 搜索：Elasticsearch 8.x
- 向量库：Qdrant
- 异步任务：Redis + Celery
- 本地 Embedding 服务：`embedding-service`（固定 `bge-m3`）

## 5. 当前真实产品边界

### 5.1 权限与账号

- 角色模型固定为 `admin` 与 `employee`
- 密级模型固定为 `L1` 到 `L4`
- 员工由管理员创建，默认账号规则为 `姓名@kms.com`
- 管理员账号为系统内置高权限账号，不走普通员工编辑/删除流程
- 认证、安全、锁定、401 清会话等规则统一以 `docs/安全基线.md` 为准

### 5.2 知识内容

- 知识结构为：知识库 -> 目录 -> 笔记 -> 附件
- 笔记正文主存储为 `content_json`
- 检索与召回使用 `content_text`
- 附件当前支持 `pdf` / `docx`
- 在线预览仅支持 `pdf`
- `docx` 统一走提示后下载

### 5.3 搜索与问答

- 搜索必须先做权限过滤
- 问答召回必须先做权限过滤
- 问答当前真实召回链路为：Elasticsearch + Qdrant hybrid recall
- 问答切分当前真实策略为：结构感知切分 + 句子级拼块 + 小重叠窗口兜底
- 问答当前使用固定模型策略，不再支持管理员自由注册/切换 chat 或 embedding 模型
- 管理员当前只保留两类 AI 相关能力：
  - 设置 `Sys Prompt`
  - 查看 QA 审计
- 问答接口当前真实形态：
  - `POST /api/v1/qa`
  - `POST /api/v1/qa/stream`
  - `GET /api/v1/qa` 仅兼容保留
- 模型或服务失败时必须结构化失败返回，不允许静默伪造答案

### 5.4 部署与运行

- 本地前端默认端口为 `3300`
- 本地后端端口为 `9002`
- Qdrant 容器端口为 `6333`，宿主机当前映射端口为 `16333`
- Embedding 服务端口为 `8001`
- 启动方式以 `docker compose` 为主

### 5.5 管理后台

- 管理后台已切换为独立 `AdminShell`，不再复用前台业务侧边栏
- 后台当前按模块拆页：
  - 总览
  - 用户管理
  - 部门管理
  - 仓库管理
  - CORS 设置
  - 认证审计
  - Sys Prompt
  - QA 审计
- 用户、部门、仓库的新增/编辑统一走弹窗，不走整页堆叠表单
- 后台总览当前只保留轻量概览信息，不做性能/Token/首字延迟等分析大盘
- 若后端尚未稳定记录对应字段，不允许在后台总览中伪造或占位展示分析指标

## 6. 当前真实功能清单

以下模块已经落地到代码，不应再被当成“待设计”：

### 6.1 已落地

- 登录、登出、当前用户、个人资料、修改密码
- CORS 白名单、安全审计、登录失败锁定、会话撤销
- 知识库、目录、笔记、附件的读取与后台管理
- 用户侧新建/删除笔记与目录
- 附件上传、替换、删除、下载、PDF 预览
- 全文检索、筛选、搜索建议、发布人员联想
- Hybrid Recall（ES + Qdrant）
- QA 严格失败返回
- QA SSE 流式接口
- 后台 Sys Prompt 配置与 QA 审计查看

### 6.2 已明确下线或禁用

- 管理员自由注册/启停/切换模型
- 用户选择默认聊天模型
- 运行时从数据库选择 chat / embedding 模型

任何 AI 继续开发时，不要把这些旧能力重新加回来，除非用户明确要求。

## 7. AI 执行约束

- 修改代码前，先以代码真实状态为准，不以旧文档习惯性推断
- 不要把历史已完成事项继续复制到 `进度.md`
- 不要在 `任务.md` 与 `进度.md` 重复写同一大段背景说明
- 不要重新引入旧的“模型治理平台”方案
- 安全相关改动必须同步核对 `docs/安全基线.md`
- 涉及中文文件排查时，PowerShell 读取优先使用：
  - `Get-Content -Encoding UTF8`

## 8. 当前待办事项

### P1：做一轮浏览器主链路回归

范围：

- 登录 / 登出 / 401 清会话
- 账号锁定提示
- 个人中心改密
- 仓库 / 目录 / 笔记 / 附件
- 搜索筛选
- QA 页面流式交互
- 管理后台用户、部门、Sys Prompt、QA 审计

验收标准：

- 核心页面无明显功能回归
- 核心跳转链路无死循环
- 关键失败场景可见且可理解

### P1：继续收口 QA 排序与引用质量

目标：

- 继续减少弱相关来源进入最终 `sources`
- 让同一文档相邻 chunk 在 QA 侧尽量合并
- 提高 `citation_status=ok` 的稳定性

当前状态：

- 低信号短文本和明显占位笔记已基本压下去
- 结构感知切分、chunk 级 hybrid recall 已生效
- 个别问题仍会保留同文档多 chunk 来源，引用完整性还不够稳

验收标准：

- 典型问题的 `sources` 尽量只保留真正相关文档
- 同文档重复来源继续下降
- 常见问答的 `citation_status` 稳定提升

### P2：补齐异步入库链路是否继续推进的决策

当前状态：

- Celery 已存在
- 但异步入库仍未形成完整生产化闭环

需要结论：

- 是继续做真正异步入库
- 还是明确维持当前同步 / 半同步方案
