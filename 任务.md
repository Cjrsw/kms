# 企业知识管理系统（KMS）MVP 任务说明

## 1. 项目目标

本项目目标是交付一个可上线的企业知识管理系统 MVP，覆盖以下核心能力：

- 用户登录与权限控制
- 知识仓库浏览、笔记阅读、笔记编辑
- PDF 与 DOCX 文档入库
- 基于 Elasticsearch 的全文检索
- 基于受权限约束知识范围的 RAG 问答
- 管理员后台中的组织、权限、仓库与审核规则管理

## 2. 已确认技术选型

- 前端框架：Next.js（App Router）
- 前端样式：Tailwind CSS
- 前端状态：Zustand
- 富文本编辑器：TipTap
- 后端框架：FastAPI
- 数据库：MySQL
- 对象存储：MinIO
- 搜索引擎：Elasticsearch 8.x
- 异步任务：Redis + Celery
- 文档解析：PyMuPDF、python-docx
- 大模型接入：Gemini API

## 4. 关键业务规则

### 4.1 权限模型

采用“角色权限 + 数据密级”双层控制。

- 角色用于控制功能权限
  - `admin`：系统管理员（内置，唯一高权限角色）
  - `employee`：员工（默认业务角色）
- 密级用于控制数据可见范围
  - `L1`：公开
  - `L2`：内部
  - `L3`：受限
  - `L4`：机密

权限判断规则：

- 用户只能搜索、查看、问答命中自身密级及以下的数据
- 用户密级不足时，不允许在检索结果中看到目标内容
- 用户密级不足时，不允许在 RAG 召回中命中目标 chunk
- 管理员可以分配员工密级（L1-L3）并维护账号状态
- 仓库、目录、笔记均可设置最小访问密级
- 笔记若未单独设置密级，则继承父目录或仓库默认密级

### 4.2 内容存储规则

- TipTap 正文主存储格式：`content_json`
- 供检索与嵌入使用的归一化纯文本：`content_text`
- Elasticsearch 只索引 `content_text`、标题、标签、作者、仓库等字段
- MinIO 存储原始附件
- MySQL 存储元数据、权限、目录结构、索引任务状态

### 4.3 检索与问答规则

- 搜索前先加权限过滤
- RAG 召回前先加权限过滤
- 参考来源至少精确到笔记
- 每个来源需保留 `repository_id`、`note_id`，便于前端跳转

### 4.4 内容审核规则

- 首版采用“规则库 + 敏感词”方案
- 触发规则后禁止发布，并返回命中原因
- 暂不引入 LLM 审核

### 4.5 认证与鉴权加固基线（2026-04-07）

- CORS 必须使用白名单模式，不允许通配。
- CORS 白名单需支持在管理员后台在线维护。
- 生产环境必须拒绝默认安全配置：
  - `SECRET_KEY=change-me` 禁止启动；
  - CORS 保持 `localhost` 默认白名单时禁止启动。
- Access Token 与前端会话 Cookie TTL 统一为 12 小时。
- 后端返回 `401` 时，前端必须走清会话流程（先注销、再清 Cookie）。
- 密码登录错误上限为 5 次，达到上限后临时锁定账号，并给前端明确反馈。
- Token 撤销机制必须同时覆盖：
  - `token_version`（账号维度批量失效）
  - `jti`（单 token 精确失效）
- 认证与鉴权行为必须有审计日志，至少覆盖登录成功/失败、鉴权失败、退出登录等关键事件。
- 详细安全基线统一维护在：
  - `docs/安全基线.md`

### 4.6 用户与组织信息规则（2026-04-07）

- 用户体系固定为“管理员 + 员工”，不再拆分多业务角色。
- 管理员创建员工时：
  - 默认账号为 `姓名@kms.com`（重名自动追加后缀）。
  - 默认密码为 `123456`。
  - 默认状态为启用。
  - 默认开启“需改密提醒”（`need_password_change=true`），但不强制首次登录改密。
- 员工资料字段至少包含：姓名、部门、职位、性别、简介、电话、邮箱、创建时间、停用时间。
- 电话与邮箱在创建员工时可留空，员工可在个人中心自行完善，管理员也可修改。
- 管理员可查询员工信息并支持按部门、姓名（关键词）、账号状态筛选；管理员不得在后台直接改员工密码。

### 4.7 目录与附件访问规则（2026-04-10）

- 仓库详情页在“当前目录”视图中，必须同时展示：
  - 当前目录的子目录列表
  - 当前目录下的笔记列表
- 仓库详情接口返回的笔记数据必须包含 `folder_id`，用于前端准确还原“目录-笔记”关系。
- 管理员创建或编辑笔记时，笔记密级不得低于父目录（若存在）或仓库设置的最小密级。
- 附件预览与下载必须走业务服务受保护接口，不得把浏览器直接重定向到 MinIO 内网地址。
- 预览能力基线：
  - 在线预览仅支持 `PDF`
  - `DOCX` 点击预览时必须给出明确提示并引导用户下载
- 登录态失效时，前端附件代理路由必须站内跳转到 `/login` 或 `/logout`，不得跳转到容器内部主机名。
- 删除能力规则补充（2026-04-11）：
  - 必须支持“笔记删除”和“目录删除”。
  - 删除接口必须做后端权限校验，不允许仅依赖前端按钮控制。
  - 非管理员删除目录时，若目录树中存在其无权访问的子目录/笔记，必须拒绝删除并返回权限错误。
  - 目录删除后，需同步处理关联索引与附件对象，避免检索脏数据与存储残留。
  - 删除入口位置统一放在左侧目录树/笔记行尾（垃圾桶按钮），不在右侧内容区单独放删除面板。

### 4.8 全文检索基线（2026-04-12）

- 检索接口采用统一分页响应，必须返回：
  - `total`
  - `page`
  - `page_size`
  - `items`
- 搜索参数至少支持：
  - `q`（关键词）
  - `repository_slug`（知识库范围）
  - `file_type`（`all|note|pdf|docx`）
  - `date_from/date_to`（更新时间范围）
  - `sort_by`（`relevance|updated_desc|updated_asc`）
  - `page/page_size`（分页）
- 搜索结果必须继续执行密级过滤，不允许绕过权限看到高密级内容。
- 搜索高亮必须在前端有明确视觉样式（不仅返回标签，还要可见）。
- 必须提供搜索建议接口（`/api/v1/search/suggest`）。
- 同义词/拼音先落地“基础别名词典”，后续再演进为可配置词库。

## 6. 系统架构

### 6.1 前端

- Next.js App Router
- 页面模块
  - 登录
  - 知识仓库
  - 仓库详情
  - 全文检索
  - 问答
  - 管理后台

### 6.2 后端

- FastAPI 提供 REST API
- Celery 负责异步入库与索引任务
- MySQL 存储业务数据
- MinIO 存储附件
- Elasticsearch 提供搜索与混合检索能力
- Redis 作为 Celery broker / backend

### 6.3 文档入库流程

1. 用户创建笔记或上传附件
2. 后端写入 MySQL 元数据与 MinIO 文件
3. 创建 `IngestionJob`
4. Celery 拉取任务并解析文件
5. 生成 `content_text`
6. 进行 chunk 切分
7. 写入 Elasticsearch
8. 更新任务状态

### 6.4 RAG 问答流程

1. 前端提交问题与仓库范围
2. 后端根据用户角色与密级生成过滤条件
3. Elasticsearch 执行召回
4. 后端构建 Prompt
5. Gemini 生成回答
6. 返回回答与来源笔记列表

## 8.1 开发环境编码注意事项

- 当前默认终端为 Windows PowerShell 5.1，直接读取 UTF-8 中文文件时可能出现乱码显示。
- 这类乱码通常是“终端读取或输出编码错误”，不代表源码文件本身已经损坏。
- 在终端中查看中文代码、文档、配置时，应优先显式指定 UTF-8 编码，例如：
  - `Get-Content -Encoding UTF8 -Raw 路径`
  - `Set-Content -Encoding UTF8 路径 内容`
- 如需排查中文文案问题，应优先以编辑器或浏览器中的实际显示结果为准，不应只根据 PowerShell 默认输出下结论。
- 后续涉及中文文案修改时，必须先确认是“终端显示问题”还是“文件内容问题”，避免误判后错误重写文本。
